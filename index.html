What is react.js?

It isn't some fancy framework - it is actually just a simple library with some useful patterns you can follow. What it is NOT is;

  • another MVC framework,
  • an all in one solution,
  • a direct alternative for something like Angular,
  • a dumping ground for business logic,
  • or some magical all round solution.

If anything, it merely gives you the 'V', or view part, of MVC. Its up to you to find something that will handle the 'M' & 'C'.

What makes up react.js?

React.js is made up out of components - and components are quite literally things you'd like to render. React then updates and render or re-render components when data changes.

You can split up your component types (I usually do) into some of the following:

  • components: These are bare bones components, usually pure and stateless without any business logic. Do not focus on the look of the component - this is about function and use.

But that's just me - you can add everything under components or remove views and add everything with more than one component under a container. That's up to you. Just remember to pick a standard and keep with it.

  • containers: These are collections of 1 or more components used to make something else. This could be forms, collections of inputs and buttons - something that isn't a complete page, but functionality wise can be separate from the page yet is made up from various components.

  • views: Think of this as a literal page - a complete rendered view. This consists of containers and components alike, just like containers and components there should not be any business logic (only render logic) in a view.

Things I wish I knew before starting with react.js

Disclaimer; this might be a long section...

Make friends with JSX

You're going to be using some JSX syntax - but don't look at JSX from the JSX website, rather have a look over at Facebook. It isn't really important functionally - but it does add some syntactic sugar.

Here's that link: https://facebook.github.io/react/docs/jsx-in-depth.html

Make small, functional focused components

Do not over-engineer and try to make a component out of everything, but if something does have its own behaviour and it makes sense to abstract it - make the component small and compact. This makes it easy to maintain, test and even extend on.

Using React.Component

This syntax for declaring a Component is most commonly used when you're making use of the various react.js life-cycles. 

								
								class YourComponent extends React.Component {
									render() return { some div element here };
								}

Using const YourComponent = props =>

If you're not making use of any life-cycles, you can declare it as a function for a simpler syntax. This usually also indicates that your component will be stateless (no state - no need for life-cycles). 

								
								const YourComponent = props => {
									return({ some div element here });
								};

Components VS Pure Components

Pure Components are similar to regular Components - with the exception that it implements the shouldComponentUpdate() lifecycle method with a shallow prop and state comparison. Thus if you're creating a Component which does not make use of state - it is more efficient to declare it as a Pure Component. You do have the option to declare it as a function, per the previous slide, but according to the React.js documentation, it is more efficient to declare it as a Pure Component.

Here's a link for more info on Pure Components: https://facebook.github.io/react/docs/react-api.html#react.purecomponent

Use ES6

It should be obvious from the previous examples - but it might be a good idea to switch over to use ES6 syntax. It gives you the ability to make use of the class declaration syntax for Components, it gives you command over Property Initialisers and even Arrow Functions - just to mention a few.

Here's a link for more info on ES6: https://babeljs.io/blog/2015/06/07/react-on-es6-plus

What other stuff do I need?

React.js, JSX, ES6 and Redux.js isn't the only two libraries and technologies you'll need to make all of this work... You'll also have to consider:

  • Babel: This literally transpiles everything and removes any build time complexities.
  • Webpack: How the hell else will you bundle your code for deploys and testing?
  • NPM: Well since you now have dependencies, why not add a dependency to manage your dependency? (INCEPTION!!!)

Babel: https://babeljs.io Webpack: https://webpack.js.org NPM: https://www.npmjs.com

If you're going react.js...

... consider that you'll need to understand JavaScript VERY well, understand the JS ecosystem and really have at least an intermediate grasp of Object Oriented and Functional Programming to use it with ease. But if you do not - don't fear! There's ton of start kits and sources out there!
